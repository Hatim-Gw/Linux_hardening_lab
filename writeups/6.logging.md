
# description: 
Setting up logging tools *rsyslog* and *auditd*, then testing them on the Ubuntu server.

# commands:
- sudo systemctl status rsyslog
- sudo tail -f /var/log/auth.log
- ssh -p [portnumber] [user]@Ubuntu_server_ip
- sudo apt install auditd audispd-plugins -y
- sudo systemctl enable --now auditd
- sudo systemclt status auditd
- sudo useradd -m auditd_tester
-sudo ausearch -k passwd_changes


# steps performed:

1. Ubuntu has the *rsyslog* tool as the default tool for reviewing or monitoring the logs, which I checked using if it is working using: 
 - sudo systemctl status rsyslog

2. then I tested the tool by trying to log in from Kali to the server using these commands
  - sudo tail -f /var/log/auth.log     [this for monitoring the logs on live]
  - (on Kali) ssh -p 48222 tsar2@172.16.237.131

*the picture shows the failed login*

3. Afterwards, I set up *auditd* tool, which is used for tracking down previous events on the system using:
  - sudo apt install auditd audispd-plugins -y
  - sudo systemctl enable --now euditd
  - sudo systemctl status auditd

 <img width="1656" height="720" alt="image" src="https://github.com/user-attachments/assets/d825e736-98fb-410e-8870-6bd62b843892" />
 *the previous picture shows hte auditd.service is enbaled, and running*

4. then added a rule for auditd to watch any changes on */etc/passwd* using:
  - sudo auditclt -w /etc/passwd -p wa -k passwd_changes

5. I added a new user to test the result of the last rule:
   - sudo useradd -m auditd_tester
   - sudo ausearch -k passwd_changes

# issues encountered:
When I restarted the server, the rule didn't work, so to fix it, I had to add the rule manually inside the **/etc/audit/rules.d/audit.rules* file using
 - sudo nano /etc/audit/rules.d/audit.rules
